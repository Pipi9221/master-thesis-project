## 四、与 Frama-C 对比

| 特性 | DG (llvm-slicer) | Frama-C |
|------|------------------|---------|
| 分析层面 | LLVM IR | C 源码 |
| 指针分析 | 流敏感/流不敏感 PTA | 值分析 |
| 控制依赖 | 标准/NTSCD | PDG |
| 输出格式 | LLVM bitcode | C 源码 |
| VLA 支持 | ⚠️ 有警告 | ❌ 导出失败 |
| 可变参数 | ✅ 正确 | ❌ 导出非法代码 |
| 函数指针 | ✅ 正确 | ❌ 未定义符号 |

### DG 优势

1. **VLA 支持**：虽然有警告，但仍能正确切片和执行
2. **可变参数**：LLVM IR 层面处理，无问题
3. **函数指针**：指针分析能正确识别调用目标
4. **多语言支持**：支持任何能编译到 LLVM IR 的语言

### Frama-C 优势

1. **源码输出**：直接输出可读的 C 代码
2. **ACSL 支持**：能处理形式化规范
3. **值分析**：更精确的数值范围分析

## 五、发现的 Bug 详细分析

### 🔴 Issue 013: Constructor/Destructor 被错误删除 (严重)

**现象**：切片后 constructor 函数被删除，只有声明没有定义

**根因**：DG 没有建模 constructor 的隐式调用

**与 Frama-C 对比**：Frama-C 有同样问题

---

### 🔴 Issue 020: memmove 源数据初始化被删除 (严重)

**现象**：memmove 的源数据初始化被删除，导致读取未初始化内存

**根因**：DG 没有正确识别 memmove 对源内存初始化的依赖

---

### 🔴 Issue 027: strcat 目标字符串依赖被删除 (严重)

**现象**：strcat 的目标字符串依赖被删除

**根因**：类似 Issue 020，字符串操作函数的内存依赖分析有问题

---

### ⚠️ Issue 007: VLA Stack 操作 (警告)

**现象**：切片 VLA 代码时出现警告
```
WARNING: Saving stack may yield unsound results!
```

**分析**：VLA 使用 `llvm.stacksave`/`llvm.stackrestore` intrinsics

**影响**：目前测试通过，但复杂场景可能有隐患

### Bug 根因分析

所有发现的 Bug 都与 **副作用分析不完整** 相关：

| Bug 类别 | 相关代码位置 | 问题描述 |
|----------|-------------|----------|
| 内存操作 | `Calls.cpp:185-256` | memcpy/memmove 只处理目标，不处理源 |
| 字符串操作 | 无特殊处理 | strcpy/strcat 未建模内存依赖 |
| 隐式依赖 | 无特殊处理 | constructor 未建模隐式调用 |
| 外部函数 | `Calls.cpp:85-152` | 未定义函数副作用处理不完整 |
