# Test19 - 数组初始化丢失分析

**日期**: 2025-01-28  
**问题**: `l_2359[5][9][5]` 数组初始化被完全删除

## 问题概述

在 Frama-C 切片过程中，`l_2359` 这个 225 元素（5×9×5）的 `const` 数组的初始化被完全删除，导致：
1. 数组未初始化（编译错误）
2. 数组中使用的依赖变量丢失

## 原始代码分析

### 数组声明和初始化

```c
// 位置: func_16() 函数中
const int32_t *l_2359[5][9][5] = {
    // 完整的 225 元素初始化
    // 每个元素的值是以下之一:
    // - (void*)0
    // - &g_832
    // - &g_2360
};
```

### 依赖的全局变量

#### g_832
```c
// T4R6YV_seed.c 第110行
static const int32_t g_832 = 0x43841F12L;
```

#### g_2360
```c
// T4R6YV_seed.c 第160行
static const int32_t g_2360 = 1L;
```

### 数组使用情况

在原始代码中，`l_2359` 数组的特定元素 `l_2359[3][0][2]` 后来被重新赋值：

```c
l_2359[3][0][2] = func_41(p_18, (p_17 , g_25), l_28[4]);
```

### 切片后的代码

```c
// source_sliced.c 和 mutant_sliced.c
static void func_16_slice_1(int16_t p_17, int8_t p_18)
{
  int32_t *l_28[9];

  // ...

}
```

### 缺失的变量

在切片代码中：
- ❌ `g_832` 完全不存在
- ❌ `g_2360` 完全不存在

## 根本原因分析

### 1. 依赖链追踪失败

**依赖链**:
```
l_2359 数组初始化
  ↓ 使用
&g_832, &g_2360  (225个元素中的大部分)
  ↓ 需要
g_832, g_2360 全局变量
  ↓ 但
g_832, g_2360 不在切片标准中
```

**结果**: Frama-C 切片工具判断 `g_832` 和 `g_2360` 不影响切片标准变量，因此将它们从切片中移除。

### 2. 数组初始化的特殊性

- `l_2359` 是一个 `const` 数组，必须在声明时初始化
- 数组初始化包含 225 个元素，每个元素都可能指向 `g_832` 或 `g_2360`
- 由于 `g_832` 和 `g_2360` 被移除，整个初始化表达式无法在切片代码中保留

### 3. 部分使用的数组元素

虽然 `l_2359[3][0][2]` 后来被赋值和使用，但：
- 数组的其他元素（224个）没有被直接使用
- 切片工具可能认为其他元素不重要
- 但 `const` 数组的初始化是整体的，无法部分保留

## 问题影响

### 编译错误
```c
int32_t const *l_2359[5][9][5];  // 未初始化的 const 数组
```
C 语言要求 `const` 变量必须在声明时初始化，因此会产生编译错误。

### 语义差异

即使修复后使用 `(void*)0` 初始化，也存在语义差异：

1. **原始行为**: 
   - 225个元素中，很多指向 `&g_832` 或 `&g_2360`
   - 这些指针值在运行时是有效的

2. **修复后的行为**:
   - 所有元素初始化为 `(void*)0`
   - 只有 `l_2359[3][0][2]` 后来被赋值

3. **潜在问题**:
   - 如果切片代码中还有其他地方访问 `l_2359` 的其他元素（虽然不在当前切片中），会出现 NULL 指针解引用
   - 控制流可能因为指针比较而改变

## 解决方案

### 方案 1: 使用 (void*)0 初始化（当前方案）

```c
int32_t const *l_2359[5][9][5] = {
    // 225个元素全部初始化为 (void*)0
    {{(void*)0,...}, ...}
};
```

**优点**:
- 满足编译要求
- 简单直接

**缺点**:
- 语义不完全准确
- 不能完全恢复原始行为

### 方案 2: 将 g_832 和 g_2360 添加到切片（理想方案）

**需要**:
1. 识别 `l_2359` 数组的初始化依赖
2. 将 `g_832` 和 `g_2360` 作为依赖变量添加到切片
3. 恢复完整的数组初始化

**优点**:
- 完全准确的切片
- 保持原始语义

**缺点**:
- 需要修改切片算法或手动干预
- 可能增加切片代码的大小

### 方案 3: 将数组改为非 const（不推荐）

```c
int32_t *l_2359[5][9][5];  // 非 const，可以后续赋值
```

**缺点**:
- 改变原始语义（原代码中确实是 const）
- 可能影响其他依赖

## 为什么切片工具会这样做？

### PDG（程序依赖图）构建的限制

1. **静态分析限制**: 
   - 切片工具通过静态分析构建 PDG
   - 对于数组初始化的依赖追踪可能不够深入

2. **切片标准驱动**:
   - 工具从切片标准（`g_823` 等）向后追踪依赖
   - `g_832` 和 `g_2360` 不在追踪路径上，被判断为"无关"

3. **数组初始化的复杂性**:
   - 225 个元素的初始化表达式非常复杂
   - 工具可能认为"整个初始化不重要"，只保留使用部分

### 依赖追踪的边界情况

这是典型的**间接依赖丢失**问题：
- `l_2359[3][0][2]` 的使用被保留 ✅
- 但 `l_2359` 的初始化依赖（`g_832`, `g_2360`）被丢失 ❌
- 导致整个初始化无法保留

## 结论

**这是一个 PDG 依赖追踪不完整的典型案例**：

1. ✅ **数据依赖**: `l_2359[3][0][2]` 的使用被正确识别
2. ❌ **初始化依赖**: `l_2359` 数组初始化依赖的 `g_832` 和 `g_2360` 未被识别
3. ❌ **完整性**: 由于缺少初始化依赖，整个数组初始化被删除

**根本原因**: Frama-C 切片工具在构建 PDG 时，对于复杂数组初始化的依赖追踪不够深入，特别是当初始化表达式涉及多个不在切片标准中的变量时。

## 建议

1. **短期修复**: 使用 `(void*)0` 初始化（当前方案）
2. **长期改进**: 改进切片工具，加强数组初始化依赖的追踪
3. **验证**: 检查修复后的代码是否还有其他依赖丢失问题
