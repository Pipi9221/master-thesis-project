# Test6 Bug 根本原因

**日期**: 2025-01-28  
**问题变量**: `g_222` (original_slice=1900855428, mutant_slice=1900855429)

## 核心问题

**依赖追踪不完整**：`l_2013` 的初始化依赖链完全丢失

## 问题表现

### 原始代码
```c
const uint8_t **l_2016 = (void*)0;
const uint8_t ** const *l_2015 = &l_2016;
const uint8_t ** const **l_2014[2][5][2] = {{{&l_2015,&l_2015},...},...};
const uint8_t ** const *** const l_2013 = &l_2014[1][1][1];  // 完整的初始化
```

### 切片代码
```c
uint8_t const ** const *** const l_2013;  // ❌ 未初始化，依赖链丢失
```

## 依赖链丢失详情

| 变量 | 原始代码 | 切片代码 | 状态 |
|------|---------|---------|------|
| `l_2016` | `= (void*)0` | ❌ 不在切片中 | **完全丢失** |
| `l_2015` | `= &l_2016` | ❌ 不在切片中 | **完全丢失** |
| `l_2014[2][5][2]` | `= {...&l_2015...}` | ❌ 不在切片中 | **完全丢失** |
| `l_2013` | `= &l_2014[1][1][1]` | ❌ 未初始化 | **初始化丢失** |

## 影响链

```
l_2013 未初始化
  ↓
l_1965 = (g_2010 == l_2013)  // 比较结果随机
  ↓
g_168 = ...(l_1965)...       // 计算结果错误
  ↓
g_222 值不一致                // 最终影响
```

## 根本原因

1. **数组索引地址计算的依赖追踪不足**
   - `&l_2014[1][1][1]` 这种复杂的地址计算
   - 切片工具未能识别需要保留 `l_2014` 数组

2. **多级指针依赖链过长**
   - 四级指针依赖：`l_2013` → `l_2014` → `l_2015` → `l_2016`
   - 可能超过依赖追踪深度限制

3. **初始化表达式的特殊处理**
   - 地址计算被视为"不重要"节点而被移除

## 结论

**这是典型的 PDG 依赖追踪不完整问题**：
- ✅ 变量使用被正确识别并保留
- ❌ 变量初始化的依赖链完全丢失
- ❌ 导致未定义行为和值不一致

## 修复建议

恢复完整的依赖链（见 `DEPENDENCY_LOSS_ANALYSIS.md` 方案 1）
