Test87 问题的简化说明
==================================================================

## 问题本质

两个语义等价的程序，Frama-C 对死代码的处理策略不一致。

## 程序结构（伪代码）

【原始程序】
```
int g_1141 = 2;

func() {
    int *p = &g_1141;
    
    if (condition_always_true) {  // 条件总是为真
        goto skip;
    }
    
    *p = new_value;  // 死代码：永远不会执行
    
    skip:
    ...
}
```

【变异程序】
```
#define Tag(x) (x)  // 恒等宏

int g_1141 = 2;

func() {
    int *p = &g_1141;
    
    if (Tag(condition_always_true)) {  // 用宏包装
        goto skip;
    }
    
    *p = Tag(new_value);  // 死代码 + 宏包装
    
    skip:
    ...
}
```

## 运行时行为

两个程序完全相同：
- g_1141 初始值 = 2
- 赋值语句永远不执行
- g_1141 最终值 = 2 ✓

## Frama-C 切片结果

【原始程序切片 g_1141】
- 策略：保守
- 结果：保留 g_1141 定义和死代码赋值
- 原因：Frama-C 无法确定死代码，选择保留

【变异程序切片 g_1141】
- 策略：激进  
- 结果：完全删除 g_1141
- 原因：Tag 宏干扰了分析，错误地认为可以删除

## 实际验证结果

运行 source_modified.c（原始切片）:
```
[DEBUG] g_1141 before assignment: 2
[DEBUG] *p_22 = 32768, will NOT execute assignment
[DEBUG] g_1141 before assignment: 2
[DEBUG] *p_22 = 32768, will NOT execute assignment
... (重复27次)
g_1141: 2
```

结论：
1. 代码块进入了 27 次 ✓
2. 赋值语句从未执行 ✓
3. g_1141 保持初始值 2 ✓

## 问题分类

**Type B: PDG 分析不完整**
- 子类型：死代码检测不稳定（Dead Code Detection Instability）
- 影响：对语义等价程序产生不同切片结果
- 根本原因：Frama-C 的死代码分析对语法包装（宏）敏感

## 理论上的正确行为

切片器应该：
1. 对语义等价的程序产生等价的切片
2. 死代码策略应该一致（全保守或全激进）
3. 不应受语法变换（恒等宏）影响
